## Problem Statement

`loom autopilot`은 4개 워커를 평균 2-3분씩 병렬 실행한다. 그동안 사용자가 보는 출력은
phase 시작 1줄, 워커 done 4줄이 전부다. 어깨너머로 보는 동료가 "지금 뭐가 돌고 있는지,
누가 빠르고 누가 느린지, 언제쯤 gate에 도달하는지" 한눈에 알 수 없다. doctor·gate
프롬프트도 무채색이라 멀티 LLM 오케스트레이션 도구의 매력이 시각으로 전달되지 않는다.

## What Makes This Cool

**Aents 사내 동료가 어깨너머로 0.5초 안에 "아 이거 멀티 LLM 오케스트레이션이 돌고
있구나"라고 정신 모델을 잡는다.** 동료가 친숙한 미감(Vercel/Vite build progress)을
빌려와 phase × 워커 매트릭스가 누적·갱신되는 모습을 보여준다. 완료된 phase는 ✓로
박혀 남고, 현재 활성 phase의 워커들은 살아있는 펄스로 표시된다. 모든 게 ANSI escape만
으로 동작 — 의존성 0, 50ms tick.

## Constraints

- 의존성 0개 유지 (Loom의 핵심 철학: "No private APIs. Just process orchestration.")
- 596 unit tests 그린 유지
- 비-TTY 환경(CI, pipe, log file 리다이렉트)에서 자동 plain log fallback
- 단일 PR로 autopilot 전체 프레임 교체 (phase·gate·doctor 동시)
- Aents 사내 동료 audience — 정보 밀도 ↑, 데모용 첫인상보다 실사용 명료함 우선

## Design Principle: Receipt-first (cross-model insight)

> Vercel/Vite를 anchor로 고른 건 meter가 아니라 *receipt*를 원한 것이다.
> 끝났을 때 Slack에 붙여넣을 build summary처럼 박혀 있는 final state가 진짜 욕망.

**먼저 final frozen state를 디자인하고, 거기서 역설계로 라이브 틱을 그린다.**
- 라이브 진행 표시는 보조이고, "끝나고 박혀 있는 그림"이 본질.
- 성공·실패·혼합 종료 후 화면을 캡처해 그대로 Slack에 붙여도 정보가 충분해야 함.
- 라이브 펄스는 receipt가 형성되어 가는 진행을 *보여줄 뿐* — 펄스 자체가 산출물이 아님.

## Premises

1. Tier 1 스코프 = Per-worker pulse + Vercel/Vite 누적 시각 + gate 색상화. 워커
   last-line preview / 토큰 rate / cost / replay는 후속 PR로 분리한다.
2. 비-TTY 환경에서 자동 fallback 필수. `process.stdout.isTTY` 가드 후 plain
   `console.log`로 회귀.
3. autopilot 전체 프레임을 단일 PR로 교체한다. 기존 console.log 호출은 모두 새
   renderer를 통해 흐른다.
4. TUI 출력은 별도 모듈로 분리하여 mock stdout writer로 ANSI 시퀀스 문자열을 캡처
   하는 unit test가 가능해야 한다.

## Cross-Model Perspective

Independent Claude subagent cold read (전체 대화 미열람, 구조화된 요약만 받음):

1. **Coolest version not considered** — "Phase as a living waveform": phase별 sparkline이 워커들의 byte velocity로 pulsing, 워커 종료 시 슬라이스 freeze, gate 도달 시 flatline + blinking caret. 4개 스톱워치 보는 것이 아니라 "build phase가 숨쉰다"가 한 글랜스에 보이는 미감. → **Tier 2 후보로 메모.**
2. **What they really want** — "Vercel/Vite over htop/btop 선택은 meter가 아닌 *receipt* 욕망." → **반영 (Receipt-first 원칙으로 격상).**
3. **50% OSS** — log-update (~3KB MIT, "redraw region without scrollback corruption"만 함, 80줄짜리 ansi-escapes 래퍼라 vendor-in 가능). → **반영 (vendor-in 형태로, deps 0 유지).**
4. **Approach A push back** — "철학은 맞지만 weekend scope에는 wrong. SIGWINCH mid-redraw, worker가 CR emit, 일요일 밤 11시에 무는 yak shaving은 철학이 아니다." 대안: pure `frame.ts`, `sink.ts`, `driver.ts(log-update vendor)`, `gate.ts`로 분리 + snapshot은 ANSI가 아니라 frame state에. → **반영 (모듈 구조 + 테스트 전략 갱신).**

## Approaches Considered

### Approach A: Zero-dep ANSI renderer
ANSI escape 시퀀스(`\x1b[F`, `\x1b[K`, `\x1b[<n>m`)를 직접 짜서 cursor up/clear/redraw.
색상도 직접 지정. ~250-350줄 신규 코드. 의존성 +0.

**Pros**: 철학 일치, 코드 100% 통제, mock writer 테스트 쉬움  
**Cons**: 터미널 폭·이모지 폭·리사이즈를 직접 처리. 오타 한 글자가 화면을 부숨.

### Approach B: picocolors + 자체 cursor renderer
색상만 picocolors(~5KB)로 위임, cursor는 직접. ~150-200줄.

**Pros**: 색상 정확성·터미널 호환성 무료. Vite 등 메이저 dev tool과 같은 라이브러리.  
**Cons**: 의존성 +1 (5KB). 철학 미세 위배.

### Approach C: ink (React TUI)
컴포넌트 기반 풀스택 TUI 프레임워크.

**Pros**: 컴포넌트 추상화, 윈도우 리사이즈 자동, 미래 확장 무료.  
**Cons**: deps 250KB+, React 도입, JSX 빌드 셋업, 철학 대폭 후퇴.

## Recommended Approach: A (Zero-dep ANSI)

### 모듈 구조 (cross-model 반영)

```
src/tui/
  ansi.ts          # ANSI escape primitives (color, cursor up/down, clear line)
  isTty.ts         # process.stdout.isTTY + NO_COLOR/FORCE_COLOR env 감지
  format.ts        # pure: 시간 / 바이트 / persona 라벨 포맷, 아이콘
  frame.ts         # PURE function: (state) => string[]. I/O 없음. snapshot 친화.
  draw.ts          # SIGWINCH·CR mid-redraw·scrollback 안전 redraw 레이어.
                   # log-update(80줄 MIT) 참조 구현을 vendor-in. deps 0 유지.
  sink.ts          # stdout writer 추상화 (mock 주입), [loom] log-above-frame 순서 보존
  driver.ts        # state machine + 250ms tick + isTTY fallback. sink 위에서 frame 호출.
  gate.ts          # colored prompt; non-TTY는 기존 readline 그대로
  index.ts         # public API: createRenderer(out, opts)

tests/tui/
  ansi.test.ts     # 색상/escape 시퀀스 정확성
  format.test.ts   # 시간/바이트/아이콘 포맷
  frame.test.ts    # 15+ canonical state → string[] snapshot. ANSI 단언 없음.
  driver.test.ts   # fake clock + mock sink 통합 1개 — redraw 시퀀스 단언
  isTty.test.ts    # NO_COLOR / FORCE_COLOR / TTY 분기
```

### 핵심 원칙

- **frame.ts는 pure function**. 입력이 `RenderState`, 출력이 `string[]`. 테스트가 ANSI 문자열이 아니라 state 구조에 걸린다.
- **draw.ts는 corner case 전담**. SIGWINCH(`process.on('SIGWINCH', ...)`), 워커가 `\r`만 emit하는 경우, scrollback 영역 침범 방지. log-update 80줄 코드를 참고해 vendor-in.
- **sink.ts는 [loom] 로그와 frame의 순서 보존**. frame은 항상 화면 맨 아래, [loom] log는 frame 위쪽에 누적.
- **gate 진입 시 driver가 frame 정지·cursor 복구**, gate.ts가 readline 처리, 종료 시 driver 재개.

### 출력 형식 (mockup)

활성 phase가 진행 중일 때:
```
loom · dark-mode
  ✓ discuss   2 personas   2.4 KB out   0:42
  ✓ plan      2 personas   3.1 KB out   1:08
  ⚬ build
      ryze    ◐ 0:38  +312 B
      zilean  ● 0:34  +198 B
  · review    queued
  · verify    queued
  · ship      queued
  · reflect   queued

  next gate ~30s
```

phase 완료 시 활성 phase 라인이 ✓로 박힘 + 워커 라인 사라짐. 다음 phase 활성화.

### Renderer 동작 원칙

- **Tick rate**: 250ms (체감상 매끈, CPU 거의 0).
- **Frame size**: 7 phase 라인 + 활성 phase의 워커 라인 N개. 매 tick마다 ANSI
  cursor up으로 N+7줄 위로 올라가 redraw. 헤더 1줄 포함.
- **로그 분리**: `[loom]` prefix 로그(error, gate decision 등)는 frame **위쪽** 별도
  영역에 누적 print. frame은 항상 화면 맨 아래.
- **fallback**: `process.stdout.isTTY === false` 또는 `NO_COLOR` env 있으면 frame
  비활성, 기존 console.log와 동일 출력.
- **gate 진입**: phase TUI 종료 후 readline. gate 프롬프트 헤더는 ANSI 박스 + 색상.
  synthesis preview는 첫 600자에서 800자로 확장 + 헤딩 색상화.
- **doctor**: smoke 실행 중인 런타임 1줄 in-place pulse. 완료된 건 ✓ 또는 MISS로 박힘.

### 색상 팔레트

| 요소 | 색 |
|---|---|
| ✓ 완료 | green (32) |
| ⚬ 활성 | cyan (36) |
| · 대기 | dim gray (90) |
| ✗ 실패 | red (31) |
| 헤더 (loom · feature) | bold |
| 메트릭 (KB, 시간) | dim (2) |
| gate 박스 테두리 | yellow (33) |

NO_COLOR 환경에서는 모두 무채색, 아이콘만 ASCII fallback (`+`, `*`, `.`, `x`).

### Renderer API

```ts
interface FrameRenderer {
  startPhase(name: LoomPhase, personas: string[]): void;
  workerProgress(persona: string, deltaBytes: number): void;
  workerDone(persona: string, status: number, signal: string | null): void;
  endPhase(name: LoomPhase, summary: { workers: number; outBytes: number; elapsedMs: number }): void;
  log(line: string): void;        // [loom] prefix logs go here
  enterGate(): void;              // suspend frame, return cursor below
  exitGate(): void;               // resume frame
  shutdown(): void;
}
```

`runner.ts`·`autopilot.ts`·`doctor.ts`의 모든 `console.log` 호출을 이 API로 라우팅.

### TTY-fallback 시 동작

```
[loom] phase: discuss  personas: ryze, zilean
[loom] done  ryze status=0
[loom] done  zilean status=0
[loom] phase discuss complete (workers=2, session=...)
```

기존 정확히 동일한 텍스트. CI 회귀 0.

## Open Questions

1. Windows 터미널(cmd, PowerShell, Windows Terminal) 호환성 어디까지? — Windows
   Terminal과 PowerShell 7+은 ANSI 지원, cmd.exe는 미지원. 현재 Loom의 Windows
   사용자가 있는지 확인 필요. 없으면 cmd.exe는 자동 fallback으로 처리.
2. 워커가 5개 이상으로 늘어날 때 (ryze + zilean + local-fast → 3개, plan은 ornn +
   orianna + hwei + zilean → 4개) frame 높이 제한? — 현재 매트릭스 보면 최대 4개.
   안전하게 8개까지 받도록.
3. `--quiet` 또는 `--no-tui` flag 추가? — 자동 isTTY 감지로 충분, 별도 flag는 후속.

## Tier 2 Memo (후속 후보)

- **Living waveform sparkline** — phase 라인 자체가 byte velocity를 표현하는 sparkline.
  활성 phase는 pulsing, 종료된 phase는 freeze된 슬라이스로 박힘. gate 도달은
  flatline + blinking caret. Receipt-first 원칙과 호환되며 시각 임팩트 추가.
  Tier 1 출시 후 사내 동료 반응 보고 결정.
- **Worker last-line preview** — 각 워커의 최신 stdout 1줄 inline. 정보 ↑이지만
  잡음 위험. Tier 1에서는 byte delta만.
- **Token rate / cost meter** — phase 종료 시 receipt에 `$0.04`, `1240 tok/s` 추가.
  매니저·PM 데모용 가치 있음. 비용 의식 angle.
- **Replay mode** — 끝난 세션을 timelapse로 재생. 데모 임팩트 큼.

## What I noticed about how you think

- **본인 도구의 audience를 사내 동료로 못박았다** — 이건 의외였다. "본인 실사용",
  "HN 데모", "enterprise 잠재고객" 다 가능했지만 "사내 동료가 어깨너머 0.5초"라는
  매우 구체적 KPI를 골랐다. 단일 인물의 이해 속도를 디자인 anchor로 쓰는 건 PM
  본능에 가깝다.
- **Vercel/Vite 미감을 anchor로 골랐다** — htop이나 Pro Tools 미터처럼 정보 밀도
  ↑ 옵션이 있었는데 "결과 동녁이 제일"인 dev tool 미감을 택했다. 본인이 사용
  하는 도구들의 미감을 import한다는 것 — Loom 자체가 "여러 도구 조합" 도구라
  취향이 일관된다.
- **합의된 Tier 1을 단일 PR 원샷으로 가겠다** — 보안 패치는 3개로 분할하는 데
  동의해 놓고 TUI는 원샷이다. 차이는 명확하다: 보안은 분할이 명시적 가치(검토·
  롤백·CHANGELOG), TUI는 분할이 손해(phase 경계에서 시각 점프). 같은 PR 분할
  원칙을 무차별 적용하지 않는 모습.
- **"의존성 정책" 옵션을 명시하라고 했을 수도 있는데 안 했다** — 4개 전제 모두
  OK로 진행. picocolors 5KB도 거부할 만큼 deps 정책이 강한 건지, 아니면 그냥
  Phase 4 기본 옵션으로 두고 거기서 결정해도 충분하다 본 건지. 답변 패턴 보면
  후자에 가깝다 — "필요할 때 결정한다" 스타일.
